# SafeRoute Database Migrations

Last reviewed: 2026-05-16

Alembic migrations are the committed source of truth for schema changes. The backend also creates tables at startup for the local SQLite hackathon path, but new model changes still need migrations.

## Current Stack

- SQLAlchemy async models live in `backend/app/models/database.py` and `backend/app/models/trips.py`.
- Alembic config lives in `backend/alembic.ini`.
- Migration environment lives in `backend/migrations/env.py`.
- Versions live in `backend/migrations/versions/`.

## Apply Migrations

```powershell
cd backend
alembic upgrade head
```

## Create a Migration

```powershell
cd backend
alembic revision --autogenerate -m "short_description"
```

Always review autogenerated migrations. Confirm indexes, foreign keys, nullability, and downgrade behavior are correct.

## Roll Back

```powershell
cd backend
alembic downgrade -1
alembic downgrade <revision_id>
```

Use `alembic downgrade base` only for disposable local databases.

## Inspect State

```powershell
cd backend
alembic current
alembic history --verbose
```

## Current Migration Themes

The migration history includes:

- initial model batch,
- identity v3 TUID/security fields,
- trips and trip stops,
- locationless SOS,
- guaranteed SOS delivery queue/audit/circuit state,
- SOS resolution fields,
- missing indexes,
- location-log table rename.

## Rules

1. Every persisted model change needs an Alembic migration.
2. Do not edit old migration files after they have been shared.
3. Do not run raw production `ALTER TABLE` outside migrations.
4. Test upgrade and downgrade for risky schema changes.
5. Keep migration docs and API docs aligned when schema changes affect contracts.
